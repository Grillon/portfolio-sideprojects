CoreMLX – Phase 1

Type : Socle IA open source
Année : 2025
Stack : FastAPI, Llama.cpp, Bash, Python, Streamlit
Rôle : Concepteur fullstack

Problème :

L’IA est partout, mais son fonctionnement reste opaque pour la majorité. Peu d’outils permettent de montrer concrètement comment elle s’exécute, s’architecture, se connecte à un système réel. La plupart masquent les rouages sous des abstractions ou des services cloud fermés.

Solution :

  • Base locale lisible et modulaire, organisée par phases
  • Phase 1 : chatbot minimal via FastAPI, modèle LLM local (llama.cpp), et interface Streamlit
  • Scripts reproductibles et documentation pas à pas pour ancrage pédagogique
  • Prépare l’extension vers CI/CD, observabilité, interchangeabilité des modèles, RAG, fine-tuning

Ce que j'ai appris :

  • Clarification des articulations backend ↔ modèle ↔ interface
  • Refonte CI minimale sans cloud, pensée pour phase suivante
  • Structuration d’un socle extensible dans un environnement contraint
  • Diffusion publique avec vidéo, slides, dépôt open source : validation complète d’une phase projet

Thugz Plugin

Type : Hackathon Web3 Unreal
Année : 2025
Stack : Unreal Engine, RPC
Rôle : Support tech & bizdev

Problème :

Les jeux classiques et web2 sont amusant mais pas les jeux web3... Les acteurs du web3 ne savent pas faire les jeux.

Solution :

  • Prototype Unreal plugin (fonctionnel)
  • Recherche Partenariat (J'ai trouvé 2 potentiel partenaires, d'accord sur le principe)
  • Pitch public et support live
  • Aucun lock et on offre toutes les possibilités, la seule limite est l'imagination

Ce que j'ai appris :

  • Nécessité d’un relai terrain solide
  • Importance des démonstrations
  • Offrir trop de possibilité fait que personne ne se sent vraiment ciblé
  • Difficulté à garder l'unité lorsqu'une idée de départ est trop large
  • Confrontation avec le vrai marché(le web3 est très mal vu par une bonne partie de la communauté gaming)
  • Un marché qui semble fermé ne l'est pas tant que cela car il y a toujours des niches interessantes

LsyncKey

Type : Prototype hardware
Année : 2019
Stack : Raspberry Pi Zero, Syncthing, Linux USB gadget
Rôle : Inventeur

Problème :

Il est parfois compliqué de garder ses données synchro entre differents PC, différents lieux, différentes plateforme. desktop, portable, PC pro...

Solution :

  • Une clé de synchronisation active qui sauvegarde et versionne en arrière plan vos fichiers important.
  • Elle est détecté comme une clé USB de données classique et aussi comme une clé logiciele.
  • Elle peut se connecter au resaux wifi et se synchroniser avec d'autres clés, pour cela il suffit de mettre 2 clé en mode synchro avec un bouton physique.
  • Le prototype est une raspi-zero avec debian et le mode gadget qui permet de la detecter comme une clé USB de données
  • Elle a plusieurs partition, l'une avec les données, une avec un point de restau et une avec l'OS.
  • Le systeme de synchronisation des données est syncthing.

Ce que j'ai appris :

  • Limites de l’UX sans interface
  • Pour des raisons inconnus le comportement changeait totalement d'un PC à un autre, même sous windows
  • Sur mon PC de dev tt fonctionnait et sur les autres c'était aleatoire.
  • L'idée me semblait simple mais chaque jours était un nouveau défi
  • J'avais des acheteurs pour le produit, qq1 m'a même payé je n'ai pas réussi à refuser et d'autres étaient pres à le faire
  • J'avais aussi trouvé des gens pret à m'aider pour le dev. D'ailleurs un ami m'a accompagné un peu sur le brainstorming.

Année : 2019

Rôle : Concepteur et animateur atelier

Initier à l’IoT avec un objet connecté simple et autonome.

Solution :

  • Prototype fonctionnel à pile sans soudure
  • Boîtier imprimé 3D, ESP8266 sur mini breadboard
  • Utilisé dans un atelier avec 12 directeurs Orange Business

  • Décalage entre prototype technique et attente produit fini
  • Rôle clé de la médiation dans un cadre innovation
